🎯 EXPLICACIÓN EN ESPAÑOL: MVC Pattern

¿QUÉ ES MVC?

MVC separa la aplicación en 3 capas independientes:

1. MODEL (Modelo)
   └─ Maneja datos y lógica
   └─ TareaModel: encapsula Tarea + métodos (save, moveToState, etc)
   └─ Comunicación con API a través de ApiService
   └─ Observer pattern para notificar cambios

2. CONTROLLER (Controlador)
   └─ Maneja eventos del usuario
   └─ Escucha clicks, inputs, drag&drop
   └─ Llama métodos del Model
   └─ Re-renderiza la View cuando el Model cambia

3. VIEW (Vista)
   └─ HTML + CSS renderizado dinámicamente
   └─ El Controller genera el HTML
   └─ Sin lógica de negocio
   └─ Responsive + accesible (a11y)

┌──────────────────────────────────────────────────────────────┐
│ FLUJO: Usuario hace click                                     │
├──────────────────────────────────────────────────────────────┤
│ 1. VIEW (HTML) captura evento                                │
│    <button onclick="click" class="btn-move">                │
│                                                              │
│ 2. CONTROLLER escucha el evento                             │
│    KanbanController.onDragStart()                           │
│                                                              │
│ 3. CONTROLLER llama MODEL                                   │
│    tarea.moveToState(newState)                             │
│                                                              │
│ 4. MODEL hace HTTP call                                     │
│    ApiService.put('/tareas/t1/move', { state: ... })      │
│                                                              │
│ 5. MODEL actualiza su estado interno                        │
│    this.data = response.data                               │
│                                                              │
│ 6. MODEL notifica listeners (Observer)                      │
│    this.notifyListeners()                                  │
│                                                              │
│ 7. CONTROLLER re-renderiza VIEW                            │
│    this.render()                                           │
│                                                              │
│ 8. Usuario ve los cambios                                   │
└──────────────────────────────────────────────────────────────┘

🏗️ PATRÓN FACTORY: Widgets Reutilizables

Problema sin Factory:
  const pieChart = createPieChart(stats);
  const overdueList = createOverdueList(stats.overdueTasks);
  const statsBox = createStatsBox(stats);
  // Código repetido en múltiples lugares

Solución con Factory:
  const widgets = [
    WidgetFactory.create('pieChart', stats),
    WidgetFactory.create('overdueList', stats.overdueTasks),
    WidgetFactory.create('taskStats', stats)
  ];
  widgets.forEach(w => container.appendChild(w));

Beneficios:
  ✓ Centralizado en un lugar
  ✓ Fácil agregar nuevos widgets
  ✓ Reutilizable en múltiples vistas
  ✓ Lógica de renderizado separada

⌨️ ACCESIBILIDAD (WCAG): Navegación por Teclado

1. Tabindex para orden de navegación
   <button tabindex="0">Click me</button>
   <input tabindex="1" />
   <div tabindex="0" role="button">...</div>

2. Arrow keys para mover tareas en Kanban
   ArrowRight: mover a siguiente estado (TODO → IN_PROGRESS)
   ArrowLeft: mover a estado anterior

3. Enter para confirmar
   <div tabindex="0" role="button" aria-label="Tarea: Implementar login">
     Presionar Enter abre detalles
   </div>

4. ARIA labels para descripción
   aria-label: Descripción de elemento
   aria-live: Para notificaciones dinámicas
   role: Semántica HTML

Test de accesibilidad:
  it('debería tener tabindex en tarjetas', () => {
    const cards = container.querySelectorAll('.task-card');
    cards.forEach(card => {
      expect(card.getAttribute('tabindex')).toBe('0');
    });
  });

🧪 TESTING: jsdom + Jest

jsdom = Simulador de DOM en Node.js (sin navegador real)

Por qué jsdom:
  ✓ Tests rápidos (no abren navegador)
  ✓ Determinísticos (sin variables del sistema)
  ✓ CI/CD compatible (servidores sin GUI)

Tests de Models:
  - Métodos de actualización (save, moveToState, etc)
  - Llamadas a API (mock de fetch)
  - Observer pattern (notificación de cambios)
  - Getters y propiedades

Tests de Controllers:
  - Renderizado correcto (elementos en DOM)
  - Event listeners adjuntos
  - Cambios de estado
  - Accesibilidad (tabindex, aria-labels)
  - Drag&drop

Ejemplo de test:
  it('debería renderizar 4 columnas Kanban', () => {
    const columns = container.querySelectorAll('.kanban-column');
    expect(columns.length).toBe(4);
  });

⚡ PERFORMANCE: <300ms para dashboard

1. Debounce para eventos frecuentes
   - Búsqueda: 300ms
   - Resize: 100ms
   - Input: 200ms

2. Throttle para dragover
   - Máximo cada 50ms
   - Evita re-renderizado excesivo

3. Lazy loading de widgets
   - Renderizar solo widgets visibles
   - SVG optimizado (pastel)
   - Canvas para gráficos complejos

4. Medición real
   const start = performance.now();
   // ... renderizar
   const elapsed = performance.now() - start;
   console.log(`⏱️ ${elapsed.toFixed(2)}ms`);
